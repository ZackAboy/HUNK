Rules:

- Screens may depend on providers, models, widgets, and simple app configuration.
- Screens should not directly call Apple HealthKit, Google Health Connect, OpenAI, Gemini, secure storage, or platform channels.
- Providers may coordinate services and expose UI-ready state.
- Services may call external SDKs, platform APIs, storage, and network clients.
- Models should stay free of UI and platform-specific dependencies.
- Widgets should be reusable and should not own business logic.
- The current shell has no service, model, or provider dependencies yet.
- New dependencies, permissions, services, background jobs, and network calls must be checked against `docs/SECURITY_AND_EFFICIENCY.md` before implementation.

## Security Architecture Principles

See `docs/SECURITY_AND_EFFICIENCY.md` for the central security reference.

- API keys should be stored only in secure storage. They should not be stored in plain shared preferences, committed files, logs, analytics, crash reports, or test fixtures.
- Health data access should be permission-scoped. Request only the metric types and access direction needed by the active feature.
- Avoid unnecessary backend transmission of raw health data.
- Prefer local processing and summarization where practical.
- Network calls must be explicit and provider-specific. AI provider services should make it clear which provider receives which prompt data.
- Future backend sync must be opt-in and documented before implementation.
- Sensitive data models should be designed to separate raw health data, summarized health data, API key material, and AI chat content.
- Debug tooling must not print raw health values, API keys, prompt payloads, or provider responses that may include private user context.

## Efficiency Architecture Principles

See `docs/SECURITY_AND_EFFICIENCY.md` for the central efficiency reference.

- Avoid unnecessary background work.
- Avoid polling when event-driven or user-triggered refresh is enough.
- Cache summaries where appropriate.
- Keep health data reads scoped by date range and metric type.
- Avoid repeatedly sending large health histories to AI providers.
- Prefer compact health summaries in prompts instead of raw data dumps.
- Design services so health reads, summary generation, and AI calls can be measured, throttled, retried intentionally, and cancelled where practical.
- Future watch, widget, or companion surfaces must use compact state and avoid frequent refreshes unless the platform requires them for a documented user-facing feature.

## Data Flow

Current data flow:

```text
No real health or AI data flow is implemented yet.
```

Intended MVP data flow:

```text
Health data source
  -> health service
  -> health summary model
  -> AI prompt
  -> chat response
  -> chat screen
```

Detailed flow:

1. Apple HealthKit or Google Health Connect provides raw health metrics after user permission.
2. A platform-aware health service reads only the permitted metric types and date ranges needed for the feature.
3. The app converts selected metrics into a compact health summary model, preferably locally.
4. The AI coach service builds a prompt that includes the compact health summary and the user's chat message.
5. The selected AI provider returns a chat response.
6. The chat screen displays the response and preserves useful conversation state.

Security and efficiency notes:

- Raw health histories should not be sent to AI providers when a compact summary is enough.
- Repeated prompt construction should reuse cached summaries when the underlying health data has not changed.
- Any new backend step in this flow must be opt-in, user-visible, and documented in `docs/DECISIONS.md` before implementation.

## Sensitive Data Boundaries

- API keys should be stored locally using secure storage when implemented.
- Raw health data should not be logged.
- Prompts containing health summaries should be treated as sensitive.
- Any future backend or sync system must be documented in `docs/DECISIONS.md` before implementation.
